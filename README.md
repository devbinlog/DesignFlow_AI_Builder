# DesignFlow AI Builder

Figma 디자인 구조를 분석하여 컴포넌트 단위로 해석하고, React + Tailwind CSS 코드 초안을 자동 생성하는 AI 기반 디자인-개발 워크플로우 시스템.

---

# 1. 프로젝트 개요

디자이너가 Figma에서 완성한 화면을 개발자가 코드로 구현하기까지, 색상 값을 직접 복사하거나, 어떤 단위가 컴포넌트인지 직접 판단하거나, 폰트 크기를 일일이 확인하는 작업 등 불필요하게 반복되는 수작업 구간이 존재합니다.

이 프로젝트는 그 구간을 단순히 자동화하는 것이 목표가 아니라, AI가 디자인 구조를 읽고 해석하여 개발자가 시작점으로 활용할 수 있는 코드 초안과 구조 분석 결과를 제공하는 것입니다.

단순한 변환 도구가 아니라 디자인을 개발 관점으로 해석하는 시스템을 만들려 했습니다.

---

# 2. 문제 정의

디자인과 개발 사이의 협업 과정에서 반복되는 문제들이 있습니다.

첫째, 디자인 해석의 불일치입니다. 디자이너가 "카드"라고 부르는 구조를 개발자가 어떻게 컴포넌트로 나누어야 하는지는 Figma에 명시되어 있지 않습니다. 레이어 구조를 보고 개발자가 직접 판단해야 합니다.

둘째, 디자인 토큰의 비체계화입니다. 피그마 파일 안에 색상, 폰트 스타일, 간격이 존재하더라도, 개발자가 이를 추출하여 변수로 정리하는 작업은 수동입니다. 동일한 색상이 여러 이름으로 산재해 있는 경우도 빈번합니다.

셋째, 코드 초안 생성의 공백입니다. Figma의 Dev Mode나 플러그인들은 CSS 값 복사 수준의 지원을 제공하지만, 실제 컴포넌트 구조를 반영한 React 코드를 생성해주지는 않습니다.

이 프로젝트는 세 가지 문제를 하나의 파이프라인으로 처리하는 구조를 설계했습니다.

---

# 3. 해결 접근 방식

핵심 아이디어는 Figma JSON을 단순히 파싱하는 것이 아니라 구조적으로 해석하는 것입니다.

## 구조 분석 단계

Figma의 노드 트리는 계층적입니다. Frame → Group → Component → Text, Shape의 계층 구조 안에는 레이아웃 의도가 암묵적으로 담겨 있습니다. 이 계층 구조를 재귀적으로 순회하며 반복 패턴, 레이아웃 방향, 배경 역할 등을 추론합니다.

## 토큰 추출 단계

노드에서 추출된 색상, 폰트 크기, 폰트 굵기, 간격, 모서리 반경 값들을 사용 빈도 기준으로 정규화합니다. 동일한 색상이 여러 노드에서 반복될 경우 하나의 토큰으로 통합되며, 각 토큰에 대응하는 Tailwind CSS 클래스 이름을 생성합니다.

## AI 해석 단계

정규화된 노드 구조와 추출된 토큰 정보를 LLM에 전달하여 세 가지 질문에 답하도록 합니다. 이 구조에서 컴포넌트로 분리되어야 하는 단위는 어디인가, 각 컴포넌트의 역할과 이름은 무엇인가, 어떤 패턴이 반복되고 있는가. 이 과정에서 단순한 코드 생성이 아니라 의미 추론이 이루어집니다.

## 코드 생성 단계

AI가 판단한 컴포넌트 목록을 기반으로 React + Tailwind CSS 코드를 파일 단위로 생성합니다. 추출된 디자인 토큰은 CSS 변수 또는 Tailwind 설정값으로 함께 생성됩니다.

# 4. 주요 기능

## Figma JSON 파싱 및 노드 정규화

설명:
- Figma Export 또는 Dev Mode에서 추출한 JSON 구조를 입력으로 받는다
- Frame, Group, Component, Text, Shape 등 노드 유형을 분류하고 계층 구조를 정규화한다
- 노드의 boundingBox, fills, strokes, effects, typography 속성을 일관된 내부 데이터 구조로 변환한다
- 이 단계에서 불필요한 중간 노드를 정리하고 분석 대상 노드를 선별한다

## 디자인 토큰 추출

설명:
- 정규화된 노드 트리에서 색상, 타이포그래피, 간격, 모서리 반경을 추출한다
- 동일한 값이 여러 노드에서 사용되는 경우 사용 횟수를 집계하여 중요도를 평가한다
- 각 토큰에 대응하는 Tailwind CSS 클래스명을 자동으로 매핑한다
- 추출된 토큰은 색상, 타이포, 간격, 반경 탭으로 분류되어 UI에 시각적으로 표시된다

## AI 기반 컴포넌트 해석

설명:
- 구조화된 노드 데이터와 디자인 토큰을 LLM 프롬프트로 변환하여 전달한다
- LLM은 컴포넌트 후보를 식별하고, 각 후보에 대한 이름, 역할, 반복 여부, 신뢰도를 반환한다
- 신뢰도 점수가 낮은 결과는 경고로 분류되어 사용자에게 별도 표시된다
- 백엔드 AI 클라이언트는 로컬 Ollama LLM을 우선 사용하고, 설정된 경우 Anthropic Claude API로 폴백한다

## React + Tailwind 코드 생성

설명:
- AI가 식별한 컴포넌트 구조를 기반으로 파일 단위의 React 컴포넌트 코드를 생성한다
- 디자인 토큰은 Tailwind 클래스로 표현되며, 코드 내 하드코딩을 최소화한다
- 생성된 파일 목록은 코드 뷰어에서 파일 탐색기 형태로 제공되며, 클립보드 복사를 지원한다

## 분석 이력 관리

설명:
- 프로젝트별로 여러 번의 분석 실행 결과를 누적하여 저장한다
- 각 분석은 pending, running, completed, failed 상태를 실시간으로 추적한다
- 분석 진행 중에는 현재 단계와 진행률을 프로그레스 바와 함께 표시한다

## 피드백 시스템

설명:
- 사용자가 AI의 해석 결과에 대해 직접 수정 사항을 기록할 수 있다
- 원본 AI 결과값과 사용자 수정값을 함께 저장하여 추후 모델 개선의 기반 데이터로 활용한다

# 5. 전체 시스템 흐름

```
사용자 입력
  └── Figma JSON 붙여넣기 또는 파일 업로드

파싱 단계
  └── 노드 유형 분류 → 계층 구조 정규화 → 속성 추출

분석 단계
  ├── 색상 / 타이포그래피 / 간격 / 반경 토큰 추출
  └── 사용 빈도 집계 및 Tailwind 매핑

AI 해석 단계
  ├── 프롬프트 구성 (구조 데이터 + 토큰 데이터)
  ├── LLM 호출 (Ollama 우선, Claude 폴백)
  └── 컴포넌트 후보 목록 + 신뢰도 반환

코드 생성 단계
  └── 컴포넌트 구조 기반 React 파일 생성

결과 저장
  └── DB에 분석 결과 저장 (토큰, 컴포넌트, 코드, 경고)

UI 표시
  ├── 디자인 토큰 탭
  ├── 컴포넌트 트리 탭
  ├── 생성 코드 뷰어 탭
  └── 경고 패널 (토글)

피드백
  └── 사용자 수정사항 기록 → 피드백 로그 저장
```

# 6. 시스템 아키텍처

## 프론트엔드

Next.js 15 App Router 기반으로 구성되며, 클라이언트/서버 컴포넌트 경계를 명확히 구분하여 설계했습다.

```
apps/web/
├── app/
│   ├── page.tsx                 # 대시보드
│   ├── projects/                # 프로젝트 목록 및 생성
│   └── analysis/[id]/           # 분석 결과 뷰어
├── components/
│   ├── layout/                  # AppShell, Sidebar, PageHeader
│   ├── panels/                  # TokenPanel, ComponentPanel
│   ├── viewers/                 # CodeViewer
│   └── common/                  # StatusBadge, ThemeToggle
├── hooks/                       # useAnalysis, useProjects
├── store/                       # analysisStore (Zustand)
└── types/                       # 공유 타입 정의
```

상태 관리는 서버 상태(React Query)와 UI 상태(Zustand)를 분리했습니다. 분석 진행 중 폴링은 React Query의 refetchInterval로 처리하며, 완료 시 자동으로 상세 데이터를 invalidate합니다.

## 백엔드

FastAPI + SQLAlchemy async 구조로 비동기 처리를 전제로 설계했습니다.

```
apps/api/
├── routers/
│   ├── projects.py              # 프로젝트 CRUD
│   ├── analysis.py              # 분석 실행 및 조회
│   └── feedback.py              # 피드백 저장
├── services/
│   ├── analysis_service.py      # 분석 오케스트레이터
│   └── ai_service.py            # AI 해석 파이프라인
├── agents/
│   ├── parser_agent.py          # Figma 노드 파싱
│   ├── token_agent.py           # 디자인 토큰 추출
│   ├── component_agent.py       # 컴포넌트 구조 분석
│   └── code_gen_agent.py        # 코드 생성
├── clients/
│   └── claude_client.py         # Ollama / Anthropic 클라이언트
└── models/                      # SQLAlchemy 모델
```

분석 실행은 비동기 태스크로 처리되어 API 응답을 블록하지 않습니다. 클라이언트는 별도 상태 폴링 엔드포인트로 진행 상황을 추적합니다.

## AI 파이프라인

세 단계의 LLM 호출로 구성됩니다.

1단계: analyze_structure — 노드 트리를 읽고 반복 패턴과 주요 구조 단위를 파악한다

2단계: name_components — 1단계 결과를 기반으로 각 컴포넌트 후보에 의미 있는 이름과 역할을 부여한다

3단계: generate_code — 최종 컴포넌트 목록과 토큰 정보를 바탕으로 React 코드 파일을 생성한다

각 단계는 독립적인 프롬프트로 구성되며, JSON 형식의 응답을 강제하여 파싱 실패를 최소화합니다.

# 7. 기술 선택 이유

## Next.js 15 App Router

페이지 단위의 서버/클라이언트 컴포넌트 분리가 필요했습니다. 분석 결과 뷰어는 실시간 폴링이 필요한 클라이언트 중심 화면이지만, 정적 레이아웃은 서버 컴포넌트로 유지하여 초기 로딩 성능을 확보했습니다. App Router의 중첩 레이아웃은 AppShell 구조를 자연스럽게 구현할 수 있었습니다.

## Tailwind CSS v4

v4에서 CSS-first 설정 방식으로 전환된 점이 프로젝트 구조에 적합했습니다. `@layer base` 안에서 `:root`와 `.dark` 변수를 함께 정의하면 다크/라이트 테마 전환을 별도 라이브러리 없이 CSS 변수만으로 처리할 수 있습니다. `@theme inline`을 활용해 Tailwind 유틸리티와 CSS 변수를 연결했습니다.

## FastAPI

비동기 분석 파이프라인을 위해 async/await 기반 프레임워크가 필요했습니다. FastAPI는 Pydantic 모델과의 통합이 자연스러워 요청/응답 직렬화 코드를 최소화할 수 있었습니다. 자동 OpenAPI 문서 생성도 개발 중 API 확인에 유용했습니다.

## SQLAlchemy async

분석 결과는 JSON 구조의 중첩된 데이터를 포함합니다. SQLAlchemy의 JSON 컬럼 타입을 사용하면 정규화 없이 분석 결과 전체를 단일 레코드에 저장할 수 있어 초기 설계를 단순하게 유지할 수 있었습니다. 비동기 세션 관리는 FastAPI의 Dependency Injection과 결합하여 커넥션 풀을 효율적으로 관리합니다.

## Zustand

React Query가 서버 상태를 담당하기 때문에 클라이언트 UI 상태는 최소화하는 것이 목표였습니다. 현재 선택된 탭, 선택된 파일, 선택된 노드와 같은 단순한 UI 상태를 전역으로 공유하는 용도로는 Zustand의 단순한 API가 적합하다고 생각했습니다.

## Ollama (로컬 LLM)

개발 단계에서 API 비용 없이 LLM을 반복 호출할 수 있는 환경이 필요했습니다. Ollama는 로컬에서 llama3.2, qwen2.5 등의 모델을 실행하며 REST API를 제공합니다. 프로덕션 환경에서는 Anthropic Claude로 전환할 수 있도록 클라이언트 계층에서 백엔드 선택을 분리했습니다.

# 9. 핵심 설계 포인트

## 컴포넌트 분리 기준

노드의 계층 깊이만으로 컴포넌트 단위를 판단하면 오류가 큽니다. 반복 등장 여부, 독립적인 boundingBox 보유 여부, 의미 있는 자식 그룹 포함 여부를 복합 기준으로 사용합니다. AI에게 단순히 "컴포넌트를 찾아라"고 지시하는 대신, 기준이 되는 구조 특성을 프롬프트에 명시하여 일관성을 높였습니다.

## 토큰 추출 기준

색상의 경우 알파 채널이 포함된 동일 색상을 다른 토큰으로 취급하면 중복이 과도하게 발생합니다. 알파 0.05 미만의 완전 투명 색상은 무시하고, 동일한 RGB 기반 색상은 사용 빈도 기준으로 대표값 하나로 통합했습니다. 간격 토큰의 경우 padding, margin, gap에서 반복적으로 등장하는 픽셀 값만 추출하여 임의적인 단일 사용 값은 제외했습니다.

## AI 활용 방식

전체 Figma JSON을 그대로 LLM에 전달하면 컨텍스트 길이를 초과하거나 중요하지 않은 데이터에 집중하는 문제가 발생합니다. 파이프라인의 각 단계에서 필요한 데이터만 추려서 전달하도록 설계했습니다. 코드 생성 단계에서는 전체 노드 트리가 아니라 AI가 이전 단계에서 결정한 컴포넌트 목록과 토큰 요약만 사용합니다.

## 코드 생성 전략

LLM이 생성하는 코드는 비결정적이므로 구조 오류 가능성이 있습니다. JSON 형식으로 파일 목록과 코드 내용을 반환하도록 응답 형식을 강제하고, 펜스 코드 블록이 포함된 경우 이를 제거하는 후처리 단계를 추가했습니다. 파싱에 실패한 경우 빈 코드 파일로 대체하여 전체 분석 결과가 유실되지 않도록 처리합니다.

# 10. 한계 및 개선 방향

## 현재 한계

Figma JSON의 구조는 파일마다 다를 수 있습니다. Auto Layout, Component Set, Variant, Boolean Property 등의 고급 Figma 기능은 현재 파서에서 완전히 지원되지 않습니다.

로컬 LLM(llama3.2 3B)은 복잡한 컴포넌트 구조에서 의미 추론 품질이 낮습니다. 신뢰도 점수가 낮은 결과가 많이 발생할 수 있으며, 이 경우 사용자의 수동 검토가 필요합니다.

생성된 코드는 시작점 수준의 초안입니다. 접근성, 반응형 처리, 인터랙션 로직은 포함되지 않습니다.

## 개선 방향

Figma REST API를 직접 연동하면 JSON 수동 복사 없이 파일 ID만으로 데이터를 가져올 수 있습니다. OAuth 연동과 함께 구현하면 사용자 경험이 크게 개선됩니다.

피드백 로그에 누적된 사용자 수정 데이터를 활용하여 프롬프트를 개선하거나, 소규모 파인튜닝 데이터로 활용할 수 있습니다. 이를 통해 특정 도메인에 특화된 해석 품질을 높일 수 있습니다.

코드 생성 시 기존 프로젝트의 컴포넌트 라이브러리 스타일을 참조하게 하면, 생성된 코드가 기존 코드베이스와 더 일관성 있게 통합될 수 있습니다.

# 11. 프로젝트를 통해 얻은 것

## 비동기 파이프라인 설계

분석 프로세스는 수초에서 수십 초가 소요되는 작업입니다. 이를 API 요청-응답 단일 사이클로 처리하는 것은 현실적이지 않습니다. 작업을 백그라운드 태스크로 분리하고, 클라이언트는 상태 폴링으로 진행 상황을 추적하는 구조를 직접 설계하면서 비동기 시스템 설계의 트레이드오프를 체험했습니다.

## LLM 통합 설계 경험

LLM을 기능의 일부로 통합하는 것은 단순 API 호출이 아닙니다. 응답 형식의 비결정성, 컨텍스트 길이 제한, 단계별 프롬프트 분리, 응답 후처리, 폴백 전략을 함께 고려해야 합니다. 각 단계의 프롬프트를 독립적으로 테스트하고 개선하는 과정에서 프롬프트 엔지니어링의 실제적인 어려움을 이해했습니다.

## CSS 아키텍처와 테마 시스템

Tailwind CSS v4의 레이어 우선순위와 CSS 변수 캐스케이딩 동작을 실제 문제를 디버깅하며 정확히 이해했습니다. `@layer base` 내부에 `:root`와 `.dark` 변수를 함께 정의해야 CSS 레이어 우선순위 충돌 없이 다크 테마가 올바르게 적용된다는 점은 공식 문서만으로는 파악하기 어려운 부분이었습니다.

## 프론트엔드-백엔드 직렬화 계약

FastAPI의 Pydantic 모델이 snake_case를 사용하고 Next.js 프론트엔드가 camelCase를 기대할 때 발생하는 직렬화 불일치 문제를 실제로 겪었습니다. `alias_generator=to_camel`과 `populate_by_name=True`를 함께 사용하는 Pydantic v2의 camelCase 직렬화 패턴을 적용하며 API 계약 설계의 중요성을 체감했습니다.

# 12. 실행 방법

## 사전 요구사항

- Node.js 18 이상
- Python 3.10 이상
- Ollama (로컬 LLM) 또는 Anthropic API Key

## 백엔드 실행

```bash
cd apps/api

pip install -r requirements.txt

cp .env.example .env
# .env에서 DATABASE_URL, ANTHROPIC_API_KEY 설정

uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

## 프론트엔드 실행

```bash
cd apps/web

npm install

# .env.local 파일 생성
echo "NEXT_PUBLIC_API_URL=http://localhost:8000" > .env.local

npm run dev
```
