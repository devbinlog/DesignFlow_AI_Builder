# 01. 범위 및 목표

## 목적

이 문서는 DesignFlow AI Builder의 **명확한 범위(scope)와 측정 가능한 목표(goals)**를 정의한다. 범위 이탈을 방지하고 모든 에이전트가 동일한 기준으로 작업할 수 있도록 경계를 명확히 한다.

---

## 핵심 결정 사항

### 범위 내 (In Scope)

#### 기능 범위
- Figma 프레임 JSON 업로드 및 파싱
- 노드 트리 정규화 (normalize)
- 디자인 토큰 추출: 색상, 타이포그래피, 간격(spacing), 반경(radius)
- 컴포넌트 후보 트리 생성
- AI 기반 컴포넌트 명명 및 해석
- React + Tailwind CSS 코드 생성
- 분석 결과 뷰어 UI
- 프로젝트 및 분석 이력 저장
- 피드백 시스템 (AI 결과 수정)

#### 기술 범위
- 랜딩 페이지 타입 프레임 (단일 페이지 디자인)
- 데스크탑 퍼스트 레이아웃
- Auto Layout 기반 구조 우선
- React + Tailwind 코드만 생성
- 단일 사용자 시스템

### 범위 외 (Out of Scope)

| 항목 | 제외 이유 |
|------|----------|
| Figma 플러그인 | 웹 앱 우선 전략 |
| 멀티 유저 / 권한 시스템 | 포트폴리오 범위 초과 |
| 실시간 Figma 동기화 | API 복잡도 과도 |
| 디자인 편집 기능 | 도구 정체성 이탈 |
| 모바일 반응형 생성 | Phase 1 외 |
| Vue, Angular 코드 생성 | 집중도 유지 |
| 디자인 피그마 컴포넌트 분석 | 고급 기능, Phase 2 |

---

## 선택 이유

범위를 좁게 설정한 이유:
1. **집중도**: 하나의 파이프라인을 완성도 있게 구현
2. **포트폴리오 효과**: 얕고 넓은 것보다 깊고 완성도 있는 것이 평가에 유리
3. **기술적 현실성**: 멀티 유저, 플러그인은 구현 복잡도가 지나치게 높음

---

## 대안 비교

### 접근 방식 A: 넓은 범위 (실패 가능성 높음)
- 모든 Figma 노드 타입 지원
- 멀티 유저 시스템 포함
- 여러 코드 타겟 지원
→ 완성도가 낮아 포트폴리오로서 가치 하락

### 접근 방식 B: 좁은 범위 (선택)
- 랜딩 페이지 프레임에 집중
- 단일 사용자
- React + Tailwind만 지원
→ 완성도 높은 시스템 구축 가능

---

## 향후 영향

현재 범위를 벗어나는 기능들은 다음 단계에서 추가 가능:
- Phase 2: 컴포넌트 미리보기 (iframe 기반)
- Phase 2: Figma REST API 직접 연동
- Phase 3: 멀티 유저 / 팀 기능
- Phase 3: 다른 디자인 도구 지원

## 측정 가능한 목표

### 기능 목표
- [ ] Figma JSON 업로드 → 토큰 추출 완료율 90% 이상
- [ ] 컴포넌트 명명 AI 신뢰도 평균 0.7 이상
- [ ] 코드 생성 결과가 TypeScript 컴파일 오류 없음
- [ ] 분석 완료까지 소요 시간 30초 이내

### 품질 목표
- [ ] Lighthouse 성능 점수 80점 이상
- [ ] API 응답 시간 (AI 제외) 500ms 이내
- [ ] 코드 커버리지 (백엔드 핵심 로직) 70% 이상

- [ ] 시스템 다이어그램 문서화 완료
- [ ] 각 기술 선택 이유 문서화
