## 1분 소개 (Elevator Pitch)

> "DesignFlow AI Builder는 Figma 디자인과 React 코드 사이의 단절을 AI로 해결하는 시스템입니다.
> 디자이너가 Figma에서 작업한 프레임의 JSON 구조를 분석하여, 색상·타이포·간격 같은 디자인 토큰을 자동으로 추출하고, AI가 컴포넌트 계층 구조를 해석하여 즉시 사용 가능한 React + Tailwind 코드를 생성합니다.
> AI는 해석 레이어일 뿐이며, 사용자가 모든 결과를 검토하고 수정할 수 있습니다."

## 문제 정의 (Problem Statement)

### 현실에서 일어나는 일

디자이너가 Figma에서 UI를 완성한 후 개발자에게 전달할 때:

1. 개발자는 디자인을 보고 처음부터 컴포넌트를 나눈다
2. 어떤 색상이 "브랜드 색상"인지, 어떤 색상이 "배경"인지 추측한다
3. 패딩, 마진, 간격 값을 일일이 Inspect 패널에서 복사한다
4. 비슷한 카드가 여러 개 있어도 반복 구조인지 판단해야 한다
5. 컴포넌트 이름을 혼자 정한다 (디자이너와 용어 불일치)

이 과정은 창의적이지 않고, 반복적이며, 실수가 발생하기 쉽다.

### 이 시스템의 해결 방식

- 디자인 토큰 자동 추출로 추측 제거
- 컴포넌트 계층 구조 자동 추론
- AI가 디자인 의도를 해석하여 의미 있는 이름 제안
- 반복 구조 자동 감지 → `.map()` 패턴 코드 생성

## 기술적 도전과 해결

### 도전 1: AI 출력의 일관성

**문제**: Claude API는 때때로 JSON이 아닌 자연어를 반환하거나, 구조가 다른 JSON을 반환함.

**해결**:
- 프롬프트에 엄격한 JSON 스키마 명세 포함
- Pydantic으로 응답 검증 및 파싱
- 검증 실패 시 재시도 로직

---

### 도전 2: Figma 노드 트리의 복잡성

**문제**: Figma JSON은 수백 개의 중첩 노드를 포함하며, AI 입력 토큰 한계를 초과함.

**해결**:
- 깊이 우선 탐색으로 중요 노드 우선 처리
- 텍스트/이미지 내용보다 구조 정보 우선 추출
- 트리 깊이 제한 옵션

---

### 도전 3: 신뢰도의 정의

**문제**: AI 결과를 사용자가 신뢰해야 할지 판단 기준이 없음.

**해결**:
- 컴포넌트 패턴 매칭 정도를 0.0~1.0 점수로 정규화
- 0.6 미만 시 경고 UI 표시
- 모든 결과에 추론 근거(reasoning) 포함

## 아키텍처 소개 포인트

### "왜 다중 에이전트 구조인가?"

> 각 에이전트가 명확한 책임(Responsibility), 입력(Input), 출력(Output)을 가지도록 설계했습니다. Product Agent가 요구사항을 정의하면, AI Agent가 프롬프트를 설계하고, Backend Agent가 파이프라인을 구현하며, Frontend Agent가 UI를 구축합니다. 이렇게 하면 각 도메인을 독립적으로 발전시킬 수 있고, 책임 경계가 명확해집니다.

### "왜 문서 우선인가?"

> 코드는 쉽게 변경되지만, 아키텍처 결정은 나중에 바꾸기 어렵습니다. 각 기술 선택의 이유와 대안을 문서로 먼저 정의하면, 구현 중 잘못된 방향으로 가더라도 빠르게 감지하고 수정할 수 있습니다.
